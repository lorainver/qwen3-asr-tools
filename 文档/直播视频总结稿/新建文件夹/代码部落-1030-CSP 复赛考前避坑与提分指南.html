CHAPTER 01最后两天的核心策略

核心结论：最后两天，算法能力的提升空间已极为有限。首要目标不是"多拿分"，而是确保会做的题不因低级错误丢分——即防止"意外爆零"。
  • 盲目刷题已无太大帮助，应将精力转向熟悉赛制规则与考试注意事项
  • 每年约 10%~20% 的学生会因低级失误导致意外爆零
  • 信奥赛的评审为纯系统评测，无阅卷老师，任何细节错误都可能导致零分
  • 只要足够细心，绝大多数低级错误完全可规避
老师寄语："最后两天你想的不是提高多少分，而是怎么去确保不要考零分。"

CHAPTER 02意外爆零：四大低级错误

以下四类错误每年都会导致大量学生爆零，且均与算法能力无关：

错误一：不熟悉 OI 赛制

部分学生对 OI 赛制的规则（如评测方式、提交规范）缺乏了解，导致操作不当。

错误二：文件输入输出流未掌握

CSP 复赛采用标准文件输入输出流，但学生平时训练中极少使用。该知识点是爆零高发区。

错误三：操作系统搞混

复赛使用 Linux 系统，而非 Windows。部分学生考前才意识到系统与平时训练环境不同，导致操作生疏。

错误四：考试时间记错

入门组为上午 8:30–12:00，提高组为下午 14:00–18:00。有学生将上下时段混淆，到场后发现考场对不上。

真实案例
文件输出流模板字符遗漏 → 四题全部爆零

一名算法能力优秀的入门组学生（模拟赛稳定 260 分以上，最高 320 分），在考前默写文件输出流模板时，循环中的某行代码始终无法正确写出。最终四道题的文件输出流全部对不上，本地测试全部通过，但实际比赛四个零全部爆零。

关键提醒：进入考场后的第一件事，应将文件输入输出流模板默写出来并注释掉，确保后续编码时不会遗漏。

CHAPTER 03文件命名与保存的致命细节

评测机逐字符匹配文件名，任何微小偏差都会导致文件无法读取，直接判零。

案例一
双重后缀名

系统隐藏了文件扩展名显示，学生手动添加 .cpp 后，实际文件名变为 apple.cpp.cpp，评测机无法读取，四题全部爆零。

案例二
保存为 .exe 而非 .cpp

应提交 .cpp 代码文件，部分学生误保存为 .exe 可执行文件，导致提交内容不正确。

案例三
文件名手动输入出错

未使用复制粘贴，手动输入文件名时出现单个字符错误，导致该题文件读取失败。

案例四
文件夹层级结构不符

实际文件需存放在以考号命名的文件夹中，且须严格按照规定的框架顺序存放。层级不正确同样会导致评测机无法读取。

正确做法（"CV大法"）：文件名务必从题目中直接复制粘贴（Control C + Control V），切忌手动逐字符输入。提交文件为 .cpp 代码文件，.exe 文件可删除。

CHAPTER 04CSP 评测机制解析

核心机制：CSP 复赛采用纯系统评测（评测机），无任何人工阅卷环节。所有评分均由固定程序自动完成。
  • 无人阅卷：没有老师查看考生试卷或代码，不存在人工纠错或"看情面"的情况
  • 严格匹配：字符级别的细节错误（多写一个点号、空格或字符）即判零分
  • 处理流程：所有考生文件统一命名后直接投入评测机，系统自动返回成绩清单
  • 出分速度：只要有评测数据，理论上当天即可出分，老师通常丢两到三次以确保稳妥
  • 核心数据：对主办方而言，最关键的资源是每道题的评测数据，有了测试数据即可立即算分
  • 参赛规模参考：以浙江为例，入门组约 2000 人、提高组约 2000 人，合计约 4000 名考生
"信奥的改卷早已是无人测评模式，不需要人去改。程序能评多少分，就是多少分。"

CHAPTER 05申诉的真实流程与意义

  • 初赛申诉：将答题卡重新送入打卡机再跑一遍，耗时约一分钟
  • 复赛申诉：将代码重新提交评测机运行一遍，同样无人工审阅
  • 需缴费：初赛和复赛的申诉均需缴纳申诉费，并非免费
  • 结果一致性：申诉结果与原评测结果通常无差异，除非存在试卷漏评
  • 系统性问题：若评测程序本身存在 bug，受影响的是大批考生而非个别学生，此类问题无需个人反馈，已有大量考生优先举报，主办方会直接发布公告并统一调整分数
残酷现实：申诉没有太大意义。该拿多少分就是多少分，调整空间极为有限。

CHAPTER 06信奥赛的公平性与五大学科对比

五大学科竞赛中，公认最公平的三个是：数学竞赛、物理竞赛、信息学竞赛。三者均为完全客观题，标准答案，纯系统评测。
学科竞赛评测方式公平性特点
信息学竞赛纯系统评测每个阶段均为评测机评测，不存在主观评分
物理竞赛系统评测为主纯客观题，标准答案
数学竞赛标准答案客观题纯客观题，无主观操作空间
其余学科含主观评分存在主观评分，有高分或低分的波动空间

信息学竞赛从 CSP → NOIP → 省选 → NOI，每个阶段均为系统评测，评分公平性是行业共识。因此很少有人质疑信奥赛的评分公正性。

CHAPTER 07骗分策略的演变与防范

骗分是指在不会做的题目中，通过输出特定值获取部分分数的行为。该策略在规则允许范围内，但近年来出题人已大幅加强防范。

早期骗分手段

  • 输出固定字符串（如 yes、no、-1）即可获得部分分数
  • 即使完全不会做题，仅输出一个字符也能拿到一定分值
  • 以前有学生和老师都会主动教授骗分技巧

当前防范措施

  • 出题人设置多组样例与测试数据，增加骗分难度
  • 题目给出的标准样例一般不在实际测试数据中出现
  • 输出结果通常为多组数据或一行数值，而非单一字符
  • 即使输出 -1，实际数据中也会混入其他值，仅输出 -1 已无法得分
补充说明：骗分属于规则允许的策略，不算违规。但当前出题方式已使骗分的收益大幅降低，不建议依赖此类策略。

CHAPTER 08入门组满分策略：会做 ≠ 拿分

核心结论：入门组一等奖的关键在于——会做的题全部拿满分，不会做的题尽可能拿部分分。做到这两点，即可领先超过 70% 的考生。
能力发挥率
≤30% 首次参赛

数据显示，首次参加 CSP 复赛的学生中，能将自身能力完整发挥出来的比例不超过 30%。约 70% 的学生即使算法学到了深搜程度，仍会因非能力因素失分。

超过 80% 的学生第二次参赛成绩会比第一次提升一百多分，核心原因并非算法能力提升，而是获得了复赛经验。

常见失分场景

失分类型具体表现后果
long long 数据类型 不确定何时使用，有人全部开启，有人全部不开启 数据溢出导致丢分
时间复杂度预估（TLE） 未提前评估程序是否会超时，思路效率低 即使代码无误，超时部分直接零分
内存溢出（RE） 未考虑特殊情况导致内存越界 实际比赛中系统不会提示 RE，只能靠经验预判
特殊情况漏判 如输出 -1 的边界条件等特殊情况未处理 本应满分的题只能拿到 60~70 分
关键差异：平时做题时，系统会直观反馈 TLE/RE 等错误；但实际 CSP 比赛中，系统不会告知你是否超时或内存溢出，完全依赖选手自身的基本功和经验预判。

CHAPTER 10考前核心行动清单

1. 默写文件输入输出流模板

考前反复练习，进入考场后第一件事即默写并注释保存，确保四道题均不遗漏。

2. 使用"CV大法"管理文件名

所有文件名从题目中复制粘贴，切忌手动输入。确认提交的是 .cpp 文件。

3. 确认操作系统为 Linux

提前在 Linux 环境下练习，熟悉操作差异，避免考场上手生。

4. 核实考试时间与考场

入门组：上午 8:30–12:00｜提高组：下午 14:00–18:00。考前再次确认准考证信息。

5. 严格遵守文件层级结构

文件必须按规定的文件夹框架和考号层级存放，不可随意调整。

6. 不要自作聪明或画蛇添足

题目要求如何保存就如何保存，如何命名就如何命名。哪怕一个百分号、一个点号的偏差都会导致零分。

铁律：信奥赛"一就是一，二就是二"。不会因为你"只差一点点"就原谅你。严格按照比赛文件要求执行，是避免意外爆零的唯一方法。
