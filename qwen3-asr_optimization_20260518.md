## 1. 前端交互与 UI 深度优化

### 1.1 文档页面复制与跳转双按钮改造
* **原痛点**：用户点击 HTML 文档链接时会直接发生页面跳转，难以直接复制 URL。
* **重构方案**：
  * 修改原有的点击跳转事件，点击 HTML 文本现在默认触发**一键复制 URL 到剪贴板**，并在右侧气泡弹出“复制成功”提示。
  * 在复制按钮右侧，优雅地新增一个具有高亮悬浮动效的 **“新窗口打开” (External Link) 图标按钮**。点击该专属按钮才会触发页面跳转，实现“复制”与“浏览”的完美解耦。
  * **关联文件**: [templates/index.html](templates/index.html) & [static/main.js](static/main.js)。

### 1.2 智库摘要卡片横向滚动条优化
* **原痛点**：AI 生成智库摘要时，若单行内容过长（如代码块、无空格长英文字符串等），会粗暴地横向撑大整个卡片甚至拉宽浏览器页面，破坏排版。
* **重构方案**：
  * 为摘要卡片的 Markdown 容器及长文本输出块引入精细的 `overflow-x: auto;` 样式，并定义了平滑的磨砂质感滚动条轨道。
  * **效果**：超长行仅在卡片内部展现左右滑动条，**保持页面整体宽度的绝对稳定与紧凑美感**。

### 1.3 思考过程分段序号逻辑确认
* **底层机制分析**：
  * 对前端大模型 `<think>` 思考折叠渲染机制进行了全面审计。
  * 确认目前的渲染逻辑会根据流式返回中的段落标志进行动态拆分。对于多段思考，序号会按照**“思考部分 1”、“思考部分 2”**进行自适应累加递增，与段落数完美匹配，具备极佳的层次感。

## 2. Ollama 实时状态与显/内存分配监控

### 2.1 极轻量资源评估
* **安全结论**：Ollama 提供的 `/api/ps` 是纯 CPU 侧的轻量元数据接口，查询仅为本地回环（`127.0.0.1`）网络通信，**不触发任何 GPU 计算，零显存开销，CPU 占用率低于 0.01%**，完全不会影响 RTX 5060 的模型推理性能，极度安全。

### 2.2 后端实时状态解析 API
* 在 `web_app.py` 中新增了异步路由 `/api/ollama_status`：
  1. 通过 `httpx.AsyncClient` 异步请求 Ollama 的本地服务端口。
  2. 精确提取出当前活跃加载的模型名称、模型总大小、**分配到显存 (VRAM) 的大小**与**常驻内存 (RAM) 的大小**。
  3. 精确计算出该模型在 GPU 显存中的**占比百分比**。

### 2.3 前端侧边栏高科技面板
* 在左侧侧边栏“AI Worker 状态”正下方，增设了 **“Ollama 运行状态”** 磨砂卡片：
  * **未启动状态**：显示灰色状态灯 `○ 未启动`。
  * **在线闲置状态**：显示科技天蓝色状态灯 `● 服务在线 (空闲)`。
  * **模型运行中状态**：状态灯亮起，卡片内淡入显示当前运行模型的完整名字（如 `qwen2.5:7b`），直观呈现“显存占比百分比（如 `100.0%`）”、具体的 CPU/GPU 内存分配（如 `CPU: 0.0G / GPU: 4.7G`），并配有一条**高饱和度蓝青色渐变的动态滑动比例填充条**，科技感与尊贵感拉满。
  * **关联文件**: [web_app.py](web_app.py)。

## 3. Uvicorn 服务 Ctrl+C 信号级秒退机制

### 3.1 鸡生蛋、蛋生鸡的生命周期死锁分析
* **原痛点**：在终端按下 `Ctrl+C` 试图终止 Web App 服务时，子进程能顺利关闭，但主进程会卡在 `INFO: Waiting for application shutdown.` 阶段长达数十秒甚至永久假死，必须返回浏览器刷新页面才能解套退出。
* **死锁成因**：
  * 主网页在开启时，为了显示显存变化，会与后台的 `/api/gpu_stats` 接口维持一条 **Server-Sent Events (SSE) 流式长连接**。
  * Uvicorn 在接收到退出信号时，会先挂起，并**强制等待所有活动的 HTTP 活跃长连接关闭**。直到连接都关了，它才触发 ASGI Lifespan 中的 `shutdown` 事件。
  * 也就是说，我们把退出标志设在 `shutdown` 事件里是没用的——因为连接不关，`shutdown` 就不会执行；而 `shutdown` 不执行，长连接就不会主动关闭。这导致系统陷入永久死锁。

### 3.2 终极解决方案：OS 信号劫持 + 3秒强制超时
1. **信号级抢占拦截 (OS Signal Hijacking)**：
   * 在服务刚刚拉起的 `@app.on_event("startup")` 阶段，主动捕获并包装 Uvicorn 的 `SIGINT` (Ctrl+C) 和 `SIGTERM` 信号。
   * **只要按下 `Ctrl+C` 的微秒瞬间，我们的拦截器会抢先一步把全局变量 `is_shutting_down` 设为 `True`**，随后再把控制权链式交还给 Uvicorn。
   * 这促使 GPU 监控流在 1s 内立即跳出循环，流式套接字主动断开，长连接完美破解。
2. **强制优雅超时参数**：
   * 在底部的 `uvicorn.run()` 中，新增了双保险配置 `timeout_graceful_shutdown=3`。即使有任何顽固连接，服务也会在 3 秒内强制被物理回收，彻底告别假死！
   * **关联文件**: [web_app.py](web_app.py)。

## 4. 轮询日志噪音深度排噪

为了让开发命令行终端像艺术品一样清爽，实施了双层过滤机制：
1. **Uvicorn 日志过滤器**：继承 `logging.Filter` 编写了自定义的 `SkipPollingFilter`，自动把 `/api/ollama_status`、`/api/worker_status` 和 `/api/gpu_stats` 这三个每 2 秒一次的请求在 `uvicorn.access` 中的记录全部静默过滤。
2. **Httpx 网络库日志屏蔽**：
   * 由于后台调用本地 Ollama 采用的是 `httpx` 库，其默认会在控制台不断打印类似 `HTTP Request: GET http://127.0.0.1:11434/api/ps "HTTP/1.1 200 OK"` 的 INFO 日志。
   * 我们显式将 `httpx` 的日志级别提升至 `WARNING`：
     ```python
     logging.getLogger("httpx").setLevel(logging.WARNING)
     ```
   * **效果**：日常 2 秒一次 of 请求日志彻底静默，控制台干干净净；一旦 Ollama 发生连接报错，错误细节依然能正常打印，确保了完美的 Debug Observability。

## 5. 深度学习框架延迟导入（Lazy Loading）零显存方案

### 5.1 2GB 隐形显存泄漏原因
* 当我们把大模型全部切换到 Ollama 后，理论上本地 Python 进程不需要吃显存。但服务启动后，物理显卡依然被吃掉了约 **2.0GB** 显存。
* **原因**：`ai_worker.py` 在启动时，会直接 `import summarizer`。而原本的 `summarizer.py` 和 `model_manager.py` 在顶部直接 `import torch` 并导入了 `transformers` 相关的大模型模块。
* **底层机制**：在 Windows + CUDA 环境下，一旦 Python 执行了 `import torch` 并触碰了大模型生态依赖，PyTorch 就会自动启动底层的 **CUDA 环境初始化与设备探针检测**。即便此时没有加载任何模型参数，英伟达显卡驱动也会自动为该进程申请分配一块 **1.5GB ~ 2.0GB 的基础 CUDA Driver Context（驱动上下文）**。这部分底噪是驱动级强制占用的，导致了显存白白浪费。

### 5.2 终极重构：按需装载（Lazy Loading）
我们将整个 AI 后端改造为完全的**“延迟加载”**机制：
* **彻底清除顶部依赖**：将 `summarizer.py` 和 `model_manager.py` 顶部的 `import torch`、`transformers` 依赖全部移出。
* **局部分支动态导入**：
  * 仅在用户切换为本地 GPU 运行模型并正式触发 `_load_model()` 时，才在内部执行局部 `import torch`。
  * 在 `chat()`、`chat_stream()` 的本地推理分支中局部导入依赖。
  * 在 `release_all()` 清理显存碎片的前一刻局部导入 `torch` 执行 empty_cache。
* **效果**：
  * 当使用默认的 Ollama 模型时，子进程 `ai_worker.py` **完全不会导入任何 PyTorch 或 Transformers 相关库**。
  * 闲置启动阶段，后台 Python 进程显存占用为 **绝对的 0.00 MB**！彻底解放了您的 RTX 5060 显卡资源！
  * **关联文件**: [summarizer.py](summarizer.py) & [model_manager.py](model_manager.py)。

## 6. 大模型本地推理方案深度对比：Ollama (GGUF) vs PyTorch (AWQ/GPTQ)

针对您的 **RTX 5060 8GB 显存** 在 Windows 11 环境下的使用场景，两种部署方案的优劣势分析如下：

| 评估维度 | Ollama 方案 (基于 llama.cpp / GGUF) | PyTorch 方案 (基于 Transformers / AWQ) |
| :--- | :--- | :--- |
| **底层架构** | **纯 C++ 开发底座**。绕过了复杂的 Python 封装层，使用高度硬件优化的 C++ / CUDA 核心算子直接驱动显卡，系统开销几乎为零。 | **Python 科学计算框架**。每次推理都要经历 Python 解释器的高维封装，具有较高的底层软件栈开销（Overhead）。 |
| **显存“公摊面积”** | **底噪开销极小** (仅约 100MB 左右)。8GB 物理显存扣除系统占用后，能腾出高达 **6.5GB** 空间用来塞模型。 | **底噪开销极其庞大**。PyTorch 框架初始化强制切走约 **2.0GB** 显存作为驱动上下文，仅剩 **4.5GB** 空间留给模型，极易导致大模型在长上下文时发生 OOM。 |
| **容错性与溢出处理** | **支持 CPU/GPU 混合分层卸载 (Offloading)**。如果模型稍微超出了显存容量，其能非常智能地将部分层交给 CPU 内存计算，**绝对不会崩溃，只会速度下降**。 | **All-or-Nothing（非全即无）策略**。只要显存溢出 1 字节，就会立即抛出 `RuntimeError: CUDA out of memory` 导致整个后台进程瞬间锁死崩溃。 |
| **Win11 系统适配** | **完美规避系统分页卡死**。Ollama 能够在应用层自主规划 CPU 内存与 GPU 显存的分块传输，完美绕过 Windows 11 WDDM 3.0+ 驱动粗暴的“Shared GPU Memory 虚拟分页机制”。 | **容易导致系统全面卡顿**。一旦显存超限，Win11 强行将显存分页交换到物理内存，导致 PyTorch 疯狂通过低速 PCIe 交换数据，**生成速度瞬间暴跌 90% 甚至导致软件假死**。 |
| **首字响应速度 (TTFT)** | **瞬时响应**。不需要执行繁重的动态张量申请和编译，打字机式输出几乎在回车按下的瞬间即刻喷涌。 | **延迟偏高**。需要为上下文申请大量的 PyTorch 动态计算图，首字输出往往有明显的几秒停顿感。 |

### 💡 架构设计结论
在 Windows 11 消费级显卡环境下，**Ollama 是目前全行业本地大模型运行最完美、最高效的载体**。

我们目前的架构设计——**默认使用 Ollama 提供大语言模型推理，本地 PyTorch 仅作为备用以及负责 Whisper 快速音频转录**，在当前技术生态下是**极其精妙、抗风险能力最强、显存效率最高的黄金组合**！
